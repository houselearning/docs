Syntax & Semantic Analysis

After Lexical Analysis turns raw code into tokens, the next critical steps in the HouseLearning compilation pipeline are **Syntax Analysis** (Parsing) and **Semantic Analysis**. These phases determine if the program is both grammatically correct and logically sound.

Phase 2: Syntax Analysis (Parsing)

The parser takes the stream of tokens and checks if they conform to the formal grammar rules of the programming language. Think of it like checking if a sentence is grammatically correct.

The Abstract Syntax Tree (AST)

The primary output of the parser is the **Abstract Syntax Tree (AST)**. The AST is a hierarchical, tree-like representation of the source code that captures the necessary structural elements of the program without including every detail from the tokens (like parentheses or semicolons).

  • **Goal:** To impose a meaningful structure on the sequence of tokens.
  • **Error Handling:** If the parser finds an error—like a missing semicolon or mismatched parentheses—it throws a **Syntax Error**, which prevents further compilation.

Phase 3: Semantic Analysis

Semantic Analysis ensures the program is logically consistent and meaningful. While the syntax checker ensures the form is correct, the semantic checker ensures the content makes sense.

Key Semantic Checks

  • **Type Checking:** Ensuring operators are applied to compatible data types (e.g., preventing the addition of a string and an integer unless explicitly supported).
  • **Declaration Checking:** Verifying that all variables, functions, and classes are properly declared before they are used.
  • **Scope Checking:** Confirming that identifiers are used within their correct defined scope.

Impact on the Piston API Call

Only after passing both Syntax and Semantic Analysis is the code considered fully validated for compilation. The validated AST is then used to generate the intermediate code that the HouseLearning Engine ultimately prepares to send to the Piston API for the final execution step. If a Syntax or Semantic Error is found, the user receives the error immediately, and no request is sent to the Piston API. This saves resources and provides faster feedback.
