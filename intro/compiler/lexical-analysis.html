Lexical Analysis: The Scanning Phase

**Lexical Analysis**, also known as scanning, is the first phase of the HouseLearning compilation process. Its job is to read the raw source code, which is a stream of characters, and group them into logical units called **tokens**.

Input, Process, and Output

The lexical analyzer (scanner) essentially turns meaningless characters into recognizable building blocks for the next stage of the compiler.

  • **Input:** A file containing raw source code (e.g., `if (x == 5)`).
  • **Process:** Stripping out comments and whitespace, recognizing patterns, and grouping characters.
  • **Output:** A stream of tokens (e.g., `[KEYWORD, "if"]`, `[PUNCTUATION, "("]`, `[IDENTIFIER, "x"]`, `[OPERATOR, "=="]`, `[LITERAL, "5"]`, `[PUNCTUATION, ")"]`).

Token Types

Tokens fall into a few primary categories that define their role in the program structure:

  • **Keywords:** Reserved words in the language (e.g., `class`, `public`, `if`, `while`).
  • **Identifiers:** Names used for variables, functions, or classes (e.g., `myVariable`, `calculateArea`).
  • **Literals:** Fixed values (e.g., numbers like `10` or strings like `"Hello"`).
  • **Operators:** Symbols that represent computations or comparisons (e.g., `+`, `-`, `=`, `==`).
  • **Punctuation:** Separators and structural symbols (e.g., `{`, `(`, `;`, `,`).

The Piston API Connection

The HouseLearning Engine uses the **Piston API** for code execution across various languages. Crucially, the code that is sent to the Piston API for compilation or execution has already successfully passed the Lexical Analysis phase. This ensures that the code submitted to Piston is structurally sound in terms of tokenization, allowing the API to focus purely on the subsequent parsing, compilation, and execution of the program.
